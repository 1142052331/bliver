# 统一个人空间与导航覆盖层治理设计

日期：2026-07-14

## 背景

当前底部“我的”入口使用新版 `MeExperience`，点击他人头像则使用旧版 `ProfileDrawer`。两者展示同一类资料，却拥有不同的视觉、布局和功能组织，已经形成两套相互分叉的个人资料体验。

`MeExperience` 每次挂载都会通过 `useProfileData` 重新请求完整资料接口。该接口除资料和足迹外，还执行近期表态、近期评论等聚合查询，前端没有缓存，因此每次打开“我的”都会等待约 5 秒后才显示完整页面。

照片墙等覆盖层由独立布尔状态控制，底部导航切换只改变主场景，没有统一清理覆盖层，导致用户已经切换到地图或动态，照片墙仍然停留在最上层。

## 目标

1. “我的”和他人资料共用同一套个人空间内容与视觉语言。
2. 缓存命中时个人空间即时显示；首次进入立即显示页面结构和已知用户资料，不再出现约 5 秒的整页阻塞。
3. 底部地图、动态、我的切换一律关闭当前覆盖层。
4. 保留本人编辑和设置能力，以及他人的私信和好友关系能力。
5. 不降低足迹可见性、访客记录和隐私保护规则。

## 非目标

- 不重新设计地图、动态或照片墙的内部视觉。
- 不改变好友关系、私信权限和足迹可见性业务规则。
- 不迁移当前没有入口的旧 `ProfilePage`。
- 不为所有服务端接口建立通用缓存层。

## 方案选择

采用共享 `ProfileExperience` 方案。本人和他人资料使用同一内容组件，通过查看上下文显示不同操作。资料数据迁移到 TanStack Query，并将首屏核心数据与昂贵的活动聚合数据拆开。

没有采用复制新版页面的方式，因为它会继续制造两套资料界面。也没有只给旧抽屉换样式，因为这无法消除重复状态、重复请求和不同交互模型。

## 组件架构

### `ProfileExperience`

共享内容组件负责：

- 头像、背景、名称、足迹数、获赞数和活跃天数；
- 概览、时间线、照片三个资料视图；
- 足迹选择和照片墙、时间线入口；
- 加载、缓存回退、空状态和失败状态；
- 根据 `isOwnProfile` 切换操作区。

本人上下文提供修改头像、背景和名称、隐私与消息设置、退出登录。查看他人时提供私信、发送好友申请、接受或拒绝申请。共享内容不直接操作全局导航，而是通过清晰的回调接口发出意图。

### 展示容器

移动端将个人空间显示为全屏目的地体验，并为底部导航预留安全区域。桌面端将同一内容放入适当宽度的右侧面板。容器只负责尺寸、进出动画、焦点和关闭行为，不复制资料内容。

现有 `MeExperience` 和 `ProfileDrawer` 在迁移完成后停止承担资料内容；能直接删除时删除，若测试迁移需要短暂兼容，则只保留薄适配层。

## 数据设计

### 查询缓存

`useProfileData` 使用 TanStack Query 管理读取状态，查询键包含目标用户和查看者身份：

```text
['profile', targetUserId, viewerId || 'guest']
```

查看者身份必须进入查询键，因为同一目标用户对本人、好友、访客和管理员返回的可见足迹与访客信息不同。默认沿用项目五分钟 `staleTime`：五分钟内重复打开直接渲染缓存；缓存过期时先显示旧数据，再在后台刷新。

资料和足迹的 Socket/UI Store 事件改为更新对应 Query 缓存。修改头像、背景或名称成功后立即更新缓存，并按需同步当前登录用户信息，避免关闭再打开后回退到旧值。

### 首屏核心数据

统一个人空间首屏只依赖：

- 公开资料字段；
- 当前查看者允许看到的最近足迹；
- 本人或管理员允许看到的访客数据。

近期表态和近期评论聚合不再阻塞统一个人空间。新界面通过显式 `view=core` 请求首屏核心数据；无参数请求暂时保留完整旧响应，避免破坏仍可能存在的旧消费者。确有消费者需要活动聚合时继续使用完整响应，后续再单独迁移。迁移时补充后端可见性测试，确保核心和完整响应的隐私过滤结果一致。

服务端保留现有 `InteractionPolicy`、`FootprintAccessService` 和位置清洗逻辑。性能优化不能绕过权限判断或直接复用未经当前查看者过滤的数据。

### 渐进渲染

进入“我的”时先使用认证缓存中的当前用户名称和头像绘制页面头部，其余区域显示稳定尺寸的骨架。核心资料返回后填充统计和足迹，不让异步内容改变整体导航位置。

缓存命中时直接显示完整资料。后台刷新失败时保留缓存并显示轻量提示；没有缓存且请求失败时显示明确的错误状态和重试按钮。

## 导航与覆盖层状态

底部导航通过统一的 `switchDestination(destination)` 入口切换场景。该入口先关闭所有临时覆盖层，再设置新的主目的地。

清理范围至少包括：

- 照片墙、时间线和资料面板；
- 通知、好友、公告和移动操作抽屉；
- 地图足迹详情、同地点面板和预览；
- 其他当前由顶层 UI Store 管理的模态或临时面板。

认证弹窗、打卡编辑或资料编辑存在未提交内容时，由对应界面注册离开保护；用户确认放弃后再完成切换。普通浏览状态不增加确认步骤。

从个人空间打开照片墙或时间线时，先关闭个人空间再打开目标覆盖层，保证同一时刻只有一个主要覆盖层。从照片墙选择足迹后关闭照片墙，切回地图并定位目标足迹。

## 错误与边界处理

- 图片加载失败时使用姓名首字头像或固定照片占位，不隐藏整个资料页。
- 用户不存在或无权查看时统一显示“暂时找不到这个资料”，不泄漏封禁或隐私状态。
- 缓存数据存在但刷新失败时保留内容，并提供非阻塞的重试入口。
- 修改资料失败时保留用户输入并显示具体错误；成功前不把临时值写入长期缓存。
- 快速连续点击不同头像时，查询键隔离不同用户，旧请求不得覆盖当前用户资料。
- 登出时移除或失效与登录查看者相关的资料缓存，防止下一位用户看到前一位用户的受限内容。

## 测试策略

### 前端

- `ProfileExperience` 在本人和他人上下文显示正确且互斥的操作。
- 重复打开相同资料使用缓存，不重复阻塞页面。
- 缓存过期后先显示旧数据并后台刷新；刷新失败仍保留旧数据。
- 快速切换目标用户不会串数据。
- 地图、动态、我的任一底部导航点击都会清理所有覆盖层。
- 从个人空间进入照片墙或时间线不会保留叠加的资料层。
- 未保存编辑触发确认，普通浏览直接切换。
- 移动端为全屏目的地，桌面端为侧边面板，二者使用同一内容组件。

### 后端

- 核心资料响应不执行近期表态和评论聚合。
- 本人、好友、访客和管理员得到的足迹范围保持现有授权语义。
- 访客列表只向本人或管理员返回。
- 私密足迹和真实坐标不会因精简接口而泄漏。
- 按需活动数据接口继续通过相同权限和位置清洗流程。

### 回归

- 资料编辑、背景和头像上传、消息设置、退出登录正常。
- 私信、好友申请、接受和拒绝正常。
- Socket 资料更新和足迹增删改能更新已缓存的个人空间。
- 照片墙选择足迹、时间线选择足迹和地图定位正常。

## 性能与验收标准

- 点击“我的”后的首个渲染帧立即出现页面结构和已知身份信息，不等待网络完成才挂载整页。
- 缓存命中时资料内容在本地交互时间内出现，不展示全页加载态。
- 温热后端环境下，核心资料请求目标为 1.5 秒内完成；Render 冷启动单独记录，不与页面自身回归混淆。
- 重复打开资料不再每次产生约 5 秒的全页等待。
- 在照片墙、资料页、通知等覆盖层打开时点击任何底部目的地，最终只显示目标主场景。
- 本人和他人个人空间拥有一致的视觉、信息结构与响应式行为。

## 实施顺序

1. 为资料读取建立 Query 缓存和核心响应契约。
2. 建立共享 `ProfileExperience`，迁移本人和他人功能。
3. 将移动端和桌面端接入共享内容容器。
4. 建立统一底部导航切换及覆盖层清理入口。
5. 迁移 Socket、编辑 mutation 和测试，移除旧资料内容实现。
6. 运行前后端测试与发布验证，并记录首开和缓存命中表现。
